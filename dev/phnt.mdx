`phnt` is a collection of Windows Native API header files that has been maintained since 2009 for the System Informer project (originally Process Hacker). It is the most up-to-date set of Native API definitions available â€” definitions have been gathered from official Microsoft header files and symbol files, as well as extensive reverse engineering.

`phnt` forms the foundation that `phlib`, `kphlib`, and all plugins build on. It is also available as a standalone project for use in any Windows user-mode or kernel-mode program that needs Native API access.

## Basic usage

Replace your usual `#include <windows.h>` with:

```c
#include <phnt_windows.h>
#include <phnt.h>
```

`phnt_windows.h` pulls in `<windows.h>` along with `NTSTATUS` values and Win32 access mask definitions, and sets up the preprocessor guards that `phnt.h` depends on. `phnt.h` then includes the individual category headers in the correct order.

<Note>
  Make sure your project targets the latest Windows SDK. The headers are designed around current SDK types; older SDKs may be missing required definitions.
</Note>

## Version targeting with PHNT_VERSION

By default `phnt.h` includes definitions for all known Windows versions (`PHNT_WINDOWS_NEW`). To restrict to a specific minimum OS version, define `PHNT_VERSION` before including `phnt.h`:

```c
#define PHNT_VERSION PHNT_WINDOWS_10
#include <phnt_windows.h>
#include <phnt.h>
```

## Definition reliability annotations

The source comments in `phnt.h` document the reliability of each definition:

| Annotation | Meaning |
|---|---|
| *(none)* | Retrieved from an official Microsoft source |
| `dbg` | Obtained from a debug message or assertion in a checked kernel build |
| `symbols, private` | From symbol files; types may be incorrect |
| `winbase` | Reconstructed from Win32-ized NT definitions in `winbase.h`; names or types may be wrong |
| `rev` | Reverse-engineered |

## Kernel-mode support

`phnt` can be used in kernel-mode drivers. Define `PHNT_MODE` before inclusion:

```c
#define PHNT_MODE PHNT_MODE_KERNEL
#include <phnt.h>
```

In kernel mode, `phnt.h` omits headers that are only valid in user space (such as `ntdbg.h`, `ntlpcapi.h`, and `ntuser.h`) and adjusts type definitions accordingly. `kphlib` uses this path â€” its base header conditionally selects between kernel and user includes:

```c
// kphlib/include/kphlibbase.h
#ifdef _KERNEL_MODE
#include <ntifs.h>
// ...
#else
#include <phnt_windows.h>
#include <phnt.h>
#endif
```

## Using phnt in a plugin

Plugins include `phnt` indirectly through `phlib`. Include the phlib umbrella header:

```c
#include <ph.h>
```

`ph.h` pulls in `phbase.h`, which includes `phnt_windows.h` and `phnt.h` in the correct order before any phlib-specific headers. You therefore get the full Native API surface automatically.

<Tip>
  If you are writing a standalone program that does not use phlib, include `phnt_windows.h` and `phnt.h` directly. If you only need a subset of APIs you can include individual category headers (such as `ntpsapi.h` for process/thread info) after `phnt_windows.h` and `phnt_ntdef.h`.
</Tip>

## Definition annotation example

Reading the header comments tells you how reliable a given definition is:

```c
// ntpsapi.h (excerpt)
typedef struct _PROCESS_BASIC_INFORMATION      // official SDK type
{
    NTSTATUS ExitStatus;
    PPEB PebBaseAddress;
    ULONG_PTR AffinityMask;
    KPRIORITY BasePriority;
    ULONG_PTR UniqueProcessId;
    ULONG_PTR InheritedFromUniqueProcessId;
} PROCESS_BASIC_INFORMATION, *PPROCESS_BASIC_INFORMATION;

// rev â€” reverse-engineered; field names and types may not be exact
typedef struct _PROCESS_PROTECTION_INFORMATION
{
    PS_PROTECTION Protection;
} PROCESS_PROTECTION_INFORMATION, *PPROCESS_PROTECTION_INFORMATION;
```

Unannotated definitions (like `PROCESS_BASIC_INFORMATION` above) come from official Microsoft sources and can be relied upon. `rev` definitions are best-effort and may change between Windows builds.
